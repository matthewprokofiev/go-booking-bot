Бот записи клиентов для салона красоты или автосервиса: клиент выбирает услугу, день и свободное время прямо в Telegram, администратор получает уведомление и управляет расписанием.

## Что делает

**Клиент:**
- «📝 Записаться» → выбор услуги → выбор дня → выбор свободного слота → подтверждение записи.
- «📋 Мои записи» — список активных записей и отмена любой из них в один тап.
- Отменённый слот сразу возвращается в продажу — его может занять другой клиент.

**Администратор** (Telegram ID указывается в `ADMIN_TG_IDS`):
- Получает уведомление о каждой новой записи и о клиентских отменах.
- `/day` — записи на выбранный день с постраничной навигацией.
- Может отменить любую запись; клиент получит уведомление об отмене.

## Стек

- **Go 1.25** — один бинарник, без внешних зависимостей в рантайме
- **[go-telegram/bot](https://github.com/go-telegram/bot)** — Telegram Bot API на context, zero-dependency
- **PostgreSQL 16** + **[pgx/v5](https://github.com/jackc/pgx)** (`pgxpool`) — без ORM
- **[goose](https://github.com/pressly/goose)** — миграции, вшитые в бинарник через `embed.FS`
- **log/slog** — структурированные логи
- **Docker + Docker Compose** — multi-stage сборка, финальный образ на alpine
- **Тесты** — table-driven unit-тесты, `pgxmock` для репозиториев, `testcontainers-go` для интеграционных

## Как запустить через docker compose

Нужен Docker с плагином Compose. Токен бота получите у [@BotFather](https://t.me/BotFather), свой Telegram ID — у [@userinfobot](https://t.me/userinfobot).

```bash
# 1. Заполните переменные окружения
cp .env.example .env
# отредактируйте .env: BOT_TOKEN и ADMIN_TG_IDS обязательны

# 2. Поднимите бота и Postgres
docker compose up -d --build

# 3. Логи
docker compose logs -f app

# 4. Остановка
docker compose down          # с сохранением данных
docker compose down -v       # вместе с данными Postgres
```

Приложение дождётся healthcheck Postgres, само применит миграции и — если база пустая — засеет 4 демо-услуги и слоты на неделю вперёд в рабочих часах (`WORK_HOURS_START`–`WORK_HOURS_END` по `BUSINESS_TZ`). Никаких ручных шагов перед первым запуском не нужно.

> На Windows команды те же, но есть нюансы с `.env`, `make` и `-race` — они разобраны в [docs/WINDOWS.md](docs/WINDOWS.md).

### Переменные окружения

| Переменная | Обязательна | По умолчанию | Описание |
|---|---|---|---|
| `BOT_TOKEN` | да | — | Токен от @BotFather |
| `DATABASE_URL` | да | — | В compose подставляется автоматически |
| `ADMIN_TG_IDS` | нет | — | Telegram ID админов через запятую |
| `BUSINESS_TZ` | нет | `Europe/Moscow` | Таймзона бизнеса |
| `WORK_HOURS_START` | нет | `10` | Начало рабочего дня |
| `WORK_HOURS_END` | нет | `20` | Конец рабочего дня |
| `APP_ENV` | нет | `local` | `local` → текстовые логи (Debug), иначе JSON (Info) |

Без `BOT_TOKEN` или `DATABASE_URL` бот не стартует и печатает понятный список проблем с ненулевым кодом выхода.

## Технические решения

### Почему транзакция против двойной брони

Два клиента могут нажать «Подтвердить» на одном слоте одновременно. Наивная схема «прочитать `is_booked` → проверить → записать» оставляет окно между проверкой и записью: оба клиента увидят слот свободным и оба получат подтверждение.

Бронирование выполняется одной транзакцией, а захват слота — атомарным выражением:

```sql
UPDATE schedule_slots SET is_booked = TRUE WHERE id = $1 AND is_booked = FALSE;
```

Проверка и захват здесь неразделимы, поэтому выиграет ровно один клиент: у него `RowsAffected = 1`, у проигравшего — `0`, и он получает `ErrSlotTaken` и предложение выбрать другое время. Инвариант проверяется интеграционным тестом, где 20 горутин штурмуют один слот и ровно одна получает запись.

Отмена — тоже одна транзакция: `status='cancelled'` в `bookings` и `is_booked=FALSE` в `schedule_slots`. Порознь эти апдейты дали бы рассинхрон: падение между ними оставило бы слот занятым навсегда.

### Почему частичный уникальный индекс

Отменённые брони остаются в таблице — история отмен нужна и бизнесу, и разбору спорных ситуаций. Поэтому колоночный `UNIQUE (slot_id)` здесь был бы багом: после первой же отмены слот стал бы навсегда непригоден для новой записи.

```sql
CREATE UNIQUE INDEX uniq_active_booking ON bookings (slot_id) WHERE status = 'active';
```

Уникальность распространяется только на активные брони: отменённых на слоте может быть сколько угодно, активная — всегда одна. База гарантирует инвариант независимо от кода приложения.

### Почему таймзона бизнеса в конфиге

Контейнер живёт в UTC, а клиенты и салон — в своём часовом поясе. Если считать «свободные слоты на сегодня» по границам суток UTC, то вечером клиент в Москве увидит слоты завтрашнего дня: с 21:00 MSK в UTC уже наступили следующие сутки.

Поэтому `BUSINESS_TZ` вынесена в конфиг, резолвится через `time.LoadLocation` один раз при старте, и вся работа со временем идёт через неё: границы дня, разбор выбранной даты, отображение слотов и записей. В БД время хранится как `timestamptz` — Postgres держит его в UTC и отдаёт с зоной, так что «сдвига на три часа» при чтении не возникает.

Отдельная тонкость: следующий день вычисляется через `AddDate(0, 0, 1)`, а не «плюс 24 часа» — в дни перевода часов сутки не равны 24 часам. Это покрыто тестом на переходе в Europe/Berlin, где сутки длятся 23 часа.
